Security for fast-moving web projects

Page Security and Repo Security

Building something quickly is useful. Building something safely is what lets it survive contact with real users, real growth, and real attackers.

Read the page security view Read the repo security view

Why this matters

A feature can look fine on a local machine and still create risk when it reaches the public web. If a project gains traction, any weak code patterns are more likely to be probed, copied, or abused.

What grows over time

The further a project moves downstream, the more corrective work is usually needed. Early cleanup is cheaper than retrofitting security into a large set of pages, scripts, and linked assets.

Good tooling helps

GitHub-based utilities can help identify unsafe patterns and suggest safer code. That is useful for teams, and also a service opportunity for people who can spot risk before it becomes an incident.

Page security

Page security means protecting the browser session itself. The main concerns are unsafe HTML insertion, inline script exposure, weak link handling, and third-party resources that may expand your attack surface.

  • Use text insertion APIs for text, not raw HTML sinks.
  • Keep scripts external where practical so CSP can be strict.
  • Add rel="noopener noreferrer" to external links that open in a new tab.
  • Prefer a CSP that allows only the resources the page actually needs.

Repo security

Repo security is broader than a single page. It covers how JavaScript, HTML, JSON, XML, generated assets, and configuration files work together across the whole repository.

  • Search for risky patterns across all file types, not only HTML.
  • Replace unsafe DOM writes in JavaScript and template code.
  • Remove secrets and browser-side token handling.
  • Keep shared headers, policies, and external links consistent.

Practical message

Basic functionality is not enough once a site becomes visible. If a project starts to get traction, developers should expect to revisit early code and make corrections before small weaknesses become public problems.

That is normal engineering work, not a failure. The goal is to make the site harder to misuse while keeping it usable, maintainable, and easy to extend.

Common fixes

Safer DOM writes

Use textContent for plain text and sanitise any HTML you genuinely need to render.

Safer event wiring

Use addEventListener instead of inline handlers.

Safer navigation

Add noopener noreferrer when opening external links in a new tab.
