A real beginner’s journey with GitHub security • April 2026

Why This Took So Much Time & Effort

This was not quick or easy. As a complete beginner repo owner, setting up basic GitHub security involved confusing screens, UI glitches (like the permanent “Configure” message), learning Pull Requests, branch protection, and many small frustrating moments.

It required patience, repeated checks, and a lot of AI guidance and encouragement. Many people give up halfway. That’s why this documented journey exists.

Why This Work Is Worth It

Even for a mostly static HTML/CSS site, basic security protects your code, your users, and your reputation. Future changes are now much safer, and GitHub shows a clean green Security overview.

What We Achieved – Current Setup

  • ✅ Tech Setup - .github folder with dependabot.yml, copilot-instructions.md, and security workflow
  • ✅ Updated - Dependabot Version Updates running (with history)
  • ✅ Visitor Security Policy - SECURITY.md file (visible on Security tab)
  • ✅ Code Validation - Branch protection on main (requires PRs + passing checks)
  • ✅ Security Agents - CodeQL / Code Scanning enabled and tested
  • ✅ Offline Scanning - Secret scanning and Dependabot alerts active (all green)

Future Checklist – Safe Workflow for Any Update

  1. Make your change → GitHub offers to create a new branch
  2. Create a Pull Request (PR) to main
  3. Wait for green checks (CodeQL, etc.) to pass
  4. Review and approve the PR yourself (see “Handling Reviewer Requests” below)
  5. Use “Squash and merge” for small changes
  6. Delete the branch after merging
  7. Check the Security tab regularly
  8. Review and merge safe Dependabot PRs when they appear

Handling Reviewer Requests (Common Situation)

What happened: After enabling branch protection, GitHub asked for a reviewer approval even though all CodeQL checks passed and it was a clean static HTML change.

You were not shown in the reviewers list, and Copilot required a paid subscription.

How we solved it:

  1. Look for the red “Bypass branch protections” button (or similar override option).
  2. Tick the confirmation checkbox if it appears.
  3. Click the red bypass button and confirm.
  4. Then merge with “Squash and merge” and delete the branch.

Better long-term fix: We edited the branch protection rule to remove the “Require approval from at least one reviewer” requirement. Now you can approve your own PRs without bypassing for normal clean changes, while still keeping protection against direct pushes and requiring checks to pass.

A Note to choose Other Beginners

If you’re following this: Yes, it really does take significant time, patience, and encouragement. You will hit confusing screens and temporary glitches. That’s completely normal.

Use AI help generously. Ask for click-by-click instructions. Celebrate every small win (first PR, first green checks, first successful merge).

The effort is worth it — for you as the repo owner and for everyone who uses or builds upon your project.
