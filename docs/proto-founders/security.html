Essential HTML Security Checks for Local Platforms

There is substantial overhead involved in conducting these tests as a positive action, and individuals are advised to use discretion about how far they want to go when testing security.

1. Validate HTML syntax

Validate HTML syntax to prevent parsing errors that could lead to unexpected behavior or vulnerabilities. Providers: W3C Markup Validation Service (current), and tools like HTMLHint (open-source, heading towards broader integration in IDEs).

2. Scan for inline JavaScript

Scan for inline JavaScript code that could execute malicious scripts locally. Providers: VirusTotal (Google-owned, current), OWASP ZAP (free, current), and Microsoft Defender (built-in for Windows, current).

3. Check external script inclusions

Check for external script inclusions from untrusted or potentially malicious sources. Providers: Burp Suite (PortSwigger, current), Acunetix (Invicti, current), and StackHawk (for DevSecOps, current).

4. Detect XSS vectors

Detect potential cross-site scripting (XSS) vectors, such as unescaped user inputs or dynamic content. Providers: OWASP ZAP (current), Snyk Code (current), and Bright Security (DAST-focused, current).

5. Look for iframe embeds

Look for iframe embeds that might load remote malicious content. Providers: Pentest-Tools Website Vulnerability Scanner (current), ImmuniWeb (AI-driven, current), and GitLab DAST (integrated in CI/CD, current).

6. Identify form elements

Identify form elements with actions pointing to suspicious or phishing URLs. Providers: Sucuri (free scanner, current), Wallarm (cloud-based, current), and Intruder.io (prioritizes high-risk issues, current).

7. Check for autoplay media

Check for autoplay media elements that could trigger exploits or unwanted resource loads. Providers: OWASP ZAP (current), Acunetix (current), and OpenVAS (open-source network scanner adaptable for web, current).

8. Scan for base64-encoded data

Scan for base64-encoded data or data URIs that might hide malware. Providers: VirusTotal (current), Microsoft Defender (current), and Trivy (Aqua Security, expanding to more web assets).

9. Verify no dangerous event handlers

Verify no dangerous event handlers (e.g., onclick with eval) that could lead to code injection. Providers: Snyk Code (current), Burp Suite (current), and Checkmarx (for code analysis, current).

10. Examine CSS rules

Examine CSS rules that could enable clickjacking or overlay attacks. Providers: OWASP ZAP (current), Bright Security (current), and Invicti (Netsparker, current).

11. Ensure proper character encoding

Ensure proper character encoding to prevent encoding-based injections or display issues. Providers: W3C Validator (current), HTMLHint (current), and Sucuri (part of broader scans).

12. Search for HTML comments

Search for HTML comments that might expose sensitive information or paths. Providers: Manual tools like grep, but automated via OWASP ZAP (current) and Snyk (heading towards more static analysis).

13. Check for meta refresh tags

Check for meta refresh tags that could redirect to malicious sites. Providers: Pentest-Tools (current), Acunetix (current), and Wallarm (current).

14. Detect phishing-like elements

Detect phishing-like elements, such as fake login forms or credential prompts. Providers: Sucuri (free for basic checks, current), ImmuniWeb (current), and Microsoft Defender (anti-phishing extensions).

15. Inspect object or embed tags

Inspect <object> or <embed> tags for risky file types or sources. Providers: OWASP ZAP (current), Burp Suite (current), and GitLab DAST (current).

16. Scan for SVG elements

Scan for SVG elements containing embedded scripts or exploits. Providers: Bright Security (current), Trivy (expanding to containerized web apps), and Snyk Code (current).

17. Validate alt texts and attributes

Validate alt texts, titles, and attributes for no embedded scripts or injections. Providers: W3C Validator (current), Acunetix (current), and Checkmarx (code-focused).

18. Look for malicious links

Look for potentially malicious links or open redirects in anchor tags. Providers: Intruder.io (current), Wallarm (current), and OpenVAS (adaptable).

19. Check for deprecated elements

Check for deprecated or vulnerable HTML elements that browsers might handle insecurely. Providers: OWASP ZAP (current), Sucuri (current), and Invicti (current).

20. Perform dynamic analysis

Perform an overall dynamic analysis (DAST) scan to simulate loading and detect runtime issues. Providers: StackHawk (current), Pentest-Tools (current), and Burp Suite (expert manual and automated, current).

If a page host is serving a community, the community may ask for added security capability. But it is likely best to avoid a coding overhead at first, and then be responsive to community sentiment.
